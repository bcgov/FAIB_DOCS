3 Simulated Annealing and the Sudoku Solver

3.1 Introduction

Why Simulated Annealing?: the goal programming in Patchworks is based on Simulated Annealing

Simulated annealing is a stochastic optimization algorithm inspired by the physical process of annealing in metallurgy, where a material is heated and then slowly cooled to reduce defects and reach a stable, low-energy state. The algorithm searches for an optimal solution by repeatedly generating small random changes to the current solution and evaluating whether they improve an objective function. Better solutions are always accepted, while worse solutions may also be accepted with a probability that depends on the increase in the objective function and the current temperature (the control parameter that determines how willing the algorithm is to accept solutions that are worse than the current solution).

This probabilistic acceptance allows the algorithm to escape local optima (A local optimum is a solution that is better than all nearby alternatives but is not necessarily the best solution among all possible solutions in the entire search space) and explore a wider region of the search space during the early stages of the optimization. As the temperature gradually decreases according to a cooling schedule(the predefined rule that determines how the temperature is gradually reduced during simulated annealing, controlling the transition from broad exploration of the search space to focused refinement of promising solutions), the likelihood of accepting worse solutions diminishes, causing the search to become increasingly focused on refining the best solutions found. This balance between exploration and exploitation makes simulated annealing an effective technique for solving complex optimization problems with many local optima (like in a forest estate model).

3.2 Simple Model

The following simple simulated annealing model begins by defining an objective function, f(x) = x^2 + 10(x), which the algorithm attempts to minimize. A random starting value of (x) is selected within a specified range, and its corresponding objective function value is calculated. During each iteration, a candidate solution is generated by adding a small random deviation to the current value of (x). The candidate is then evaluated using the objective function and compared with the current solution. If the candidate produces a lower objective function value, it is always accepted. If it is worse, it may still be accepted with a probability determined by the Metropolis acceptance criterion, (P = e^{-Delta/T}), where (Delta) is the increase in the objective function and (T) is the current temperature. This probabilistic acceptance of inferior solutions allows the algorithm to escape local minima and continue exploring the search.

As the optimization progresses, the temperature is gradually reduced using a cooling schedule, causing the algorithm to become increasingly selective (read “greedy”) about accepting worse solutions. Early in the search, the high temperature encourages exploration by allowing occasional uphill moves, whereas later iterations behave more like a greedy optimization algorithm, accepting primarily improving solutions. Throughout the optimization, the algorithm records the current solution, objective function value, temperature, and whether each proposed move was accepted. These data are then used to visualize the optimization process, including the change in objective function over time, the trajectory of the search variable, and the path taken across the objective function landscape. Although this example optimizes a single variable, it illustrates the same principles used in more complex applications, such as Sudoku solving below, where the search space consists of many decision variables and the objective is to minimize the number of constraint violations.

#-------------------------------------------------------
# Simulated Annealing Example
# Objective: Minimize f(x) = x^2 + 10*sin(x)
#-------------------------------------------------------

# Objective function
objective <- function(x) {
  x^2 + 10 * sin(x)
}


#-----------------------------
# Initialize algorithm
#-----------------------------
current_x <- runif(1, -10, 10)
current_cost <- objective(current_x)

temperature <- 10
cooling_rate <- 0.99
iterations <- 1000

# Store results
results <- tibble(
  iteration = 1,
  x = current_x,
  cost = current_cost,
  temperature = temperature,
  accepted = TRUE
)

#-----------------------------
# Simulated Annealing
#-----------------------------
for (i in 2:iterations) {
  
  # Generate a nearby candidate solution
  candidate_x <- current_x + rnorm(1, mean = 0, sd = 1)
  candidate_cost <- objective(candidate_x)
  
  # Difference in objective value
  delta <- candidate_cost - current_cost
  
  accept <- FALSE
  
  # Accept better solutions
  if (delta <= 0) {
    
    accept <- TRUE
    
  } else {
    
    # Accept worse solutions with probability
    probability <- exp(-delta / temperature)
    
    if (runif(1) < probability) {
      accept <- TRUE
    }
  }
  
  # Update current solution if accepted
  if (accept) {
    current_x <- candidate_x
    current_cost <- candidate_cost
  }
  
  # Reduce temperature
  temperature <- temperature * cooling_rate
  
  # Save iteration
  results <- add_row(
    results,
    iteration = i,
    x = current_x,
    cost = current_cost,
    temperature = temperature,
    accepted = accept
  )
}

#-----------------------------
# Best solution found
#-----------------------------
best <- results %>%
  filter(cost == min(cost))

print(best)

3.3 Sudoku Solver

Who doesn’t like Sudoku….but sometime you have to guess and/or cheat.

The following is based on a python algorithm depicted here.

The Sudoku solver is a R based simulated annealing algorithm that treats the puzzle as an optimization problem rather than solving it through exhaustive search.

It begins with a partially filled 9×9 grid where zeros represent empty cells, and it identifies which values are fixed and which can be changed. The solver then constructs an initial candidate solution by randomly filling each 3×3 block so that every block contains the numbers 1 through 9 exactly once. This guarantees that block constraints are always satisfied, leaving only row and column violations to be corrected (see constraint decomposition definition at the bottom). It evaluates the quality of a solution using a cost (error) function that counts duplicate values in rows and columns, with the goal of driving this error down to zero.

To improve the grid, the solver uses simulated annealing, a probabilistic optimization technique inspired by physical cooling processes.At each step, it generates a new candidate solution by swapping two non-fixed values within a randomly chosen block, preserving block validity while altering row and column structure. If the swap reduces the number of conflicts, it is accepted; if it increases the error, it may still be accepted with a probability that depends on the current “temperature” parameter.

This temperature gradually decreases over time, allowing broad exploration early on and more focused, “greedy” improvements later. Through repeated iterations of proposing, evaluating, and conditionally accepting changes, the solver progressively reduces conflicts until it reaches a valid Sudoku solution or stops after a set number of iterations.

The seed for the Sudoku solver has been set to 999 to yield a solved Sudoku Solution…not all randomized Sudoku boards can be solved within the set iteration limit. To experiment with various Sudoku boards set the seed to NULL: set.seed(999) in the code chunk below.

3.3.1 Code Notes:

3.3.1.1 INITIAL SUDOKU SETUP

The generateFullSudoku function creates a complete, valid Sudoku grid by first building a structured base pattern and then randomizing it by shuffling rows, columns, and the numbers 1–9 while preserving all Sudoku rules. The createPuzzle function takes this full solution and randomly replaces a specified fraction of cells with zeros, creating a partially filled puzzle. Finally, generateSudokuWithZeros combines these steps to produce a new Sudoku puzzle with empty cells each time it is run.

3.3.1.2 Initial Sigma

The estimateInitialSigma function estimates a suitable starting “temperature” for simulated annealing by sampling how much the Sudoku error changes under random moves. It repeatedly performs random swaps within blocks, records the resulting error values, and collects these into a sample set. Finally, it returns the standard deviation of those errors, which serves as a data-driven measure of how large typical changes are and helps scale the initial acceptance probability appropriately.

3.3.1.3 DEFINE 3x3 BLOCKS

The getBlocks function constructs and returns a list of all nine 3×3 subgrids (blocks) in the Sudoku by generating the row–column coordinate pairs for each block. Each element of the list represents one block and contains the indices of the cells belonging to that block, making it easy to access and manipulate them. This is important because the solver relies on operating within these blocks—such as swapping values—while preserving block validity, which simplifies the problem by ensuring that only row and column conflicts need to be optimized.

3.3.1.4 GENERATE A NEIGHBOR (STATE TRANSITION)

The swapCells function repeatedly selects two random cells within a given 3×3 block and swaps their values, but only if both cells are not fixed (i.e., not part of the original puzzle). This ensures that the original clues remain unchanged while still allowing the solver to modify the grid. It is important because it generates new candidate solutions for the simulated annealing process while preserving block validity.

# Set-Up

set.seed(999) 
generateFullSudoku <- function() {
  base <- matrix(0, 9, 9)
  
  # Simple valid base pattern
  for (i in 1:9) {
    for (j in 1:9) {
      base[i,j] <- ((i-1)*3 + floor((i-1)/3) + j - 1) %% 9 + 1
    }
  }
  
  # Randomly shuffle rows within blocks
  for (b in 0:2) {
    rows <- sample(1:3) + b*3
    base[(b*3+1):(b*3+3), ] <- base[rows, ]
  }
  
  # Randomly shuffle columns within blocks
  for (b in 0:2) {
    cols <- sample(1:3) + b*3
    base[, (b*3+1):(b*3+3)] <- base[, cols]
  }
  
  # Shuffle numbers 1–9 globally
  perm <- sample(1:9)
  base <- matrix(perm[base], 9, 9)
  
  return(base)
}

createPuzzle <- function(fullSudoku, emptyFraction = 0.6) {
  puzzle <- fullSudoku
  
  # Total cells to remove
  n_remove <- round(81 * emptyFraction)
  
  # Random positions
  positions <- sample(1:81, n_remove)
  
  puzzle[positions] <- 0
  
  return(puzzle)
}


generateSudokuWithZeros <- function(emptyFraction = 0.6) {
  full <- generateFullSudoku()
  puzzle <- createPuzzle(full, emptyFraction)
  return(puzzle)
}

# IDENTIFY FIXED (GIVEN) CELLS
# ==========================================================
# Converts original puzzle into a mask:
# 1 = fixed (cannot change)
# 0 = free (can be modified)
fixSudokuValues <- function(sudoku) {
  (sudoku != 0) * 1
}



estimateInitialSigma <- function(sudoku, fixed, blocks) {
  samples <- c()
  tmp <- sudoku
  
  for (i in 1:20) {
    block <- blocks[[sample(1:9,1)]]
    tmp <- swapCells(tmp, fixed, block)
    samples <- c(samples, calcErrors(tmp))
  }
  
  return(sd(samples))
}


# ==========================================================
# COST FUNCTION (OBJECTIVE FUNCTION)

# The following calculates the total number of rule violations in 
# the Sudoku grid by summing 
# how many duplicate values exist across all rows and columns.
# ==========================================================
# This measures how "wrong" the Sudoku is.
#
# Idea:
# - Each row and column should contain numbers 1–9 exactly once.
# - If duplicates exist, we count errors.
#
# Lower cost = better solution
# Perfect solution = cost = 0

calcErrors <- function(sudoku) {
  errors <- 0
  for (i in 1:9) {
    # Count duplicates in row i and column i
    errors <- errors +
      (9 - length(unique(sudoku[i,]))) +   # row errors
      (9 - length(unique(sudoku[,i])))     # column errors
  }
  return(errors)
}


#Define 3 X3 Blocks
# ==========================================================
# Returns list of 9 blocks, each containing coordinates

getBlocks <- function() {
  blocks <- list()
  idx <- 1
  for (br in 0:2) {
    for (bc in 0:2) {
      cells <- expand.grid(
        (1:3) + br*3,
        (1:3) + bc*3
      )
      blocks[[idx]] <- as.matrix(cells)
      idx <- idx + 1
    }
  }
  return(blocks)
}


# ==========================================================
# 6. INITIAL RANDOM SOLUTION
# ==========================================================
# Fill each 3x3 block with missing numbers randomly
#
# IMPORTANT DESIGN CHOICE:
# - Each 3x3 block is always VALID (contains numbers 1–9)
# - Only rows and columns may contain conflicts

fillBlocks <- function(sudoku, blocks) {
  for (block in blocks) {
    values <- sudoku[block]
    
    # Find numbers missing in this block
    missing <- setdiff(1:9, values)
    
    # Fill empty cells randomly with missing numbers
    zeros <- which(values == 0)
    values[zeros] <- sample(missing)
    
    sudoku[block] <- values
  }
  return(sudoku)
}

# ==========================================================
# GENERATE A NEIGHBOR (STATE TRANSITION)

# We create a "new candidate solution" by:
# - Picking a random 3x3 block
# - Swapping two NON-FIXED cells inside it
#
# This ensures:
# - Block remains valid
# - Only row/column structure changes

swapCells <- function(sudoku, fixed, block) {
  repeat {
    idx <- sample(1:nrow(block), 2)
    c1 <- block[idx[1],]
    c2 <- block[idx[2],]
    
    # Only swap if both are NOT fixed
    if (fixed[c1[1],c1[2]] == 0 &&
        fixed[c2[1],c2[2]] == 0) {
      
      # Swap values
      temp <- sudoku[c1[1],c1[2]]
      sudoku[c1[1],c1[2]] <- sudoku[c2[1],c2[2]]
      sudoku[c2[1],c2[2]] <- temp
      
      return(sudoku)
    }
  }
}

3.4 Simulated Annealing Solver

The solveSudoku function applies a simulated annealing optimization process to solve a Sudoku puzzle by iteratively reducing conflicts in rows and columns while preserving valid 3×3 blocks (Constraint Decomposition). It begins by identifying fixed cells, generating an initial valid state by randomly filling each block, and estimating an initial temperature (sigma) that controls how freely the algorithm explores different configurations. During each iteration, it proposes a new candidate solution by swapping two non-fixed cells within a random block, evaluates whether the change improves the error (number of conflicts), and accepts it either if it is better or with a probability that allows occasional acceptance of worse solutions to escape local minima. As the algorithm progresses, the temperature gradually decreases (cooling), making the search increasingly focused and less random, while continuously tracking the best solution found. The process continues until either a conflict-free Sudoku (error = 0) is reached or a maximum number of iterations is exceeded, at which point the best available solution is returned.

3.4.1 Code Notes:

3.4.1.1 Simulated Annealing Parameters

The line sigma <- estimateInitialSigma(sudoku, fixed, blocks) computes a data-driven starting temperature for the simulated annealing process by estimating how much the error typically changes under random swaps. This is important because it scales the acceptance probability so that early in the search the algorithm neither accepts too many bad moves (too random) nor rejects them all (gets stuck). By setting sigma based on the actual problem landscape, it improves the solver’s balance between exploration and convergence, making the optimization more stable and effective.

The Sudoku Solver result is embedded in the bottom of the executed code chunk

solveSudoku <- function(sudoku) {
  
  # Identify fixed cells
  fixed <- fixSudokuValues(sudoku)
  
  # Get block definitions
  blocks <- getBlocks()
  
  # Create INITIAL VALID STATE
  sudoku <- fillBlocks(sudoku, blocks)
  
  #sigma <- 2            # "temperature" (controls randomness)
  
  sigma <- estimateInitialSigma(sudoku, fixed, blocks)

  cooling <- 0.995      # how fast temperature decreases
  maxIter <- 100000      # maximum attempts
  
  # Track best solution seen so far
  best <- sudoku
  bestScore <- calcErrors(sudoku)
  
  # -----------------------------
  # MAIN OPTIMIZATION LOOP
  # -----------------------------
  for (iter in 1:maxIter) {
    
    # Step 1: Generate candidate solution
    block <- blocks[[sample(1:9,1)]]
    newSudoku <- swapCells(sudoku, fixed, block)
    
    # Step 2: Evaluate current vs new state
    currentCost <- calcErrors(sudoku)
    newCost <- calcErrors(newSudoku)
    
    # Difference in cost
    delta <- newCost - currentCost
    
    # -----------------------------------------------------
    # Step 3: Acceptance rule (CORE of Simulated Annealing)
    # -----------------------------------------------------
    #
    # Case 1: If new solution is BETTER (delta <= 0)
    #         -> always accept
    #
    # Case 2: If WORSE (delta > 0)
    #         -> accept with probability:
    #
    #            exp(-delta / temperature)
    #
    # This allows the algorithm to:
    # - Escape local minima
    # - Explore wider solution space early on
    #
    if (delta <= 0 || runif(1) < exp(-delta/sigma)) {
      sudoku <- newSudoku
    }
    
    # Step 4: Track best solution found
    currentCost <- calcErrors(sudoku)
    if (currentCost < bestScore) {
      best <- sudoku
      bestScore <- currentCost
    }
    
    # Step 5: COOLING (reduce randomness over time)
    #
    # Early: high temperature → more random exploration
    # Late:  low temperature → greedy refinement
    sigma <- sigma * cooling
    
    # Step 6: Stop if solved
    if (bestScore == 0) {
      cat("Solved at iteration:", iter, "\n")
      return(best)
    }
  }
  
  cat("Stopped without perfect solution.\n")
  return(best)
}

3.5 3D plot of Sudoku loss landscape

This script snippet analyzes and visualizes how the Sudoku solver explores the solution space over time.

After the optimization runs are complete, the script investigates the structure of the search process by applying Principal Component Analysis (PCA) to the recorded Sudoku states.Before performing PCA, any cells whose values never changed are removed since they provide no information about the search dynamics. PCA projects the remaining high-dimensional Sudoku configurations into a three-dimensional space, allowing the optimization trajectories to be visualized. The script then identifies potential lock-in points, where the solver begins making very little progress, by detecting when the average rate of error reduction falls below a specified threshold over a sliding window of iterations.

These points are highlighted on the visualizations. Three complementary visual outputs are produced: a two-dimensional PCA plot showing the optimization landscape colored by error with each run’s trajectory overlaid and lock-in points marked; an animated version illustrating how each optimization path evolves over time; and an interactive three-dimensional PCA plot that enables exploration of the trajectories from different viewing angles. Together, these visualizations provide insight into how simulated annealing navigates the Sudoku solution landscape, whether different runs converge toward similar regions, and where the search begins to stagnate.

3.5.1 Why use PCA

Principal Component Analysis (PCA)is a statistical technique that simplifies complex datasets by finding the main patterns of variation within the data. Imagine you have data with many variables—in this case, each Sudoku board (Each candidate solution is a different Sudoku board configuration) is represented by 81 cell values,meaning every board exists as a point in an 81-dimensional space. Humans cannot visualize spaces with that many dimensions, so PCA transforms the data into a new set of variables, called principal components, that capture as much of the important information as possible while using far fewer dimensions.

PCA works like finding the best viewing angle for a complicated object. Imagine looking at a tangled ball of string. From one direction it might look like a messy blob,but if you rotate it to the right angle, you can clearly see its overall shape. PCA automatically finds these “best angles” by identifying the directions in which the data varies the most. The first principal component (PC1) captures the greatest amount of variation, the second principal component (PC2) captures the next greatest amount (while being independent of PC1), and the third principal component (PC3) captures the next largest remaining variation.

For the Sudoku optimization algorithm, each recorded Sudoku board represents one point in a very high-dimensional search space. PCA projects these points into two or three dimensions while preserving as much of their relative structure as possible. This allows the optimization process to be visualized as a path through a landscape. States that appear close together inthe PCA plot are generally similar Sudoku configurations, while distant points represent substantially different boards. As the simulated annealing algorithm progresses, its trajectory through the PCA space illustrates how it explores different regions of the search space before converging toward a solution.

It is important to remember that PCA does not change or improve the data — it only provides a simpler representation of it. Because the projection reduces many dimensions to just two or three, some information is inevitably lost. However, if the first few principal components explain most of the variation in the data, the resulting visualization still provides an accurate picture of the overall search behavior. This makes PCA a valuable tool for understanding optimization algorithms, identifying clusters of similar solutions, observing convergence, and detecting when the search becomes trapped in a local optimum.

# fixSudokuValues, getBlocks, fillBlocks,

# ---------------------------
# 1. Collect samples (single run)
# ---------------------------
collectSamples <- function(startingSudoku, steps = 1000) {
  fixed <- fixSudokuValues(startingSudoku)
  blocks <- getBlocks()
  
  s <- fillBlocks(startingSudoku, blocks)
  
  sigma <- 2
  cooling <- 0.995
  
  states <- matrix(NA, nrow = steps, ncol = 81)
  errors <- numeric(steps)
  
  for (i in 1:steps) {
    block <- blocks[[sample(1:9, 1)]]
    new <- swapCells(s, fixed, block)
    
    d <- calcErrors(new) - calcErrors(s)
    
    if (d <= 0 || runif(1) < exp(-d / sigma)) {
      s <- new
    }
    
    sigma <- sigma * cooling
    
    states[i, ] <- as.vector(s)
    errors[i] <- calcErrors(s)
  }
  
  list(states = states, errors = errors)
}

detect_lockin <- function(errors, window = 30, threshold = 0.01) {
  for (i in 1:(length(errors) - window)) {
    slope <- abs(errors[i + window] - errors[i]) / window
    if (slope < threshold) return(i)
  }
  return(NA)
}

3.5.3 Constraint Decomposition and the Sudoku Solver

Constraint decomposition is the process of breaking a large, complex set of constraints into smaller, simpler groups that can be handled independently or more efficiently. Rather than checking every rule simultaneously, the problem is divided into components, each responsible for enforcing a subset of the constraints.

Using Sudoku as an example, there are three types of constraints:

  1. Every row must contain the numbers 1–9 exactly once.
  2. Every column must contain the numbers 1–9 exactly once.
  3. Every 3×3 block must contain the numbers 1–9 exactly once.

Instead of treating these as one large, interconnected constraint satisfaction problem, the simulated annealing algorithm decomposes the problem by enforcing the block constraints from the very beginning.

Here’s how that works:

  • The fillBlocks() function fills each 3×3 block so that it already contains the numbers 1–9 exactly once.
  • The swapCells() function only swaps values within the same block.
  • Because values never leave their block, every 3×3 block remains valid throughout the entire optimization.

As a result, the algorithm never has to worry about violating block constraints—they are guaranteed to be satisfied. The only constraints that remain to optimize are the rows and columns, which are measured by calcErrors().This decomposition dramatically reduces the complexity of the search:

Without decomposition: Every move could violate rows, columns, and blocks simultaneously, making the search space much larger. With decomposition: The block constraints are permanently satisfied, so the algorithm only searches among Sudoku boards that already have valid blocks. The optimization focuses solely on fixing row and column conflicts.

A useful analogy is organizing a library. Suppose you have 900 books that need to be sorted by genre, author, and publication year. Solving all three sorting criteria at once would be difficult. Instead, you might first separate the books into genres and never allow books to move between genre sections. Once the genre constraint is fixed, you only need to arrange the books within each genre by author and year. By solving one part of the problem first, the remaining problem becomes much simpler.

In the Sudoku solver, the 3×3 blocks are analogous to the genres: they are solved first and then held fixed. The simulated annealing algorithm only needs to optimize the remaining row and column constraints. This is why the approach is called constraint decomposition—the original Sudoku constraint satisfaction problem is decomposed into a fixed subproblem (valid blocks) and an optimization subproblem (valid rows and columns).
