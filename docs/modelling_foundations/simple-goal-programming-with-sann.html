4 Simple Goal Programming with SANN

4.1 Introduction

Why Goal Programming with SANN?: Patchworks uses Goal Programming based on a Simulated Annealing (SANN) algorithm. The following provides a simple overview of the technique.

Goal programming using simulated annealing is a multi-objective optimization approach that seeks a solution that best satisfies several competing management goals while respecting specified constraints. Instead of optimizing a single objective, goal programming minimizes the weighted deviations from predefined target values, allowing modellers and decision-makers to assign different priorities to objectives such as timber production, economic return, and habitat conservation. Simulated annealing is then used as the search algorithm to explore the solution space by iteratively modifying candidate solutions and evaluating their weighted goal deviations. Early in the search, the algorithm accepts both improving and some non-improving solutions, enabling it to escape local optima and explore a wider range of alternatives. As the search progresses, the temperature is gradually reduced, making the algorithm increasingly selective until it converges on a solution that provides a near-optimal compromise among the competing goals while satisfying all management constraints.

4.2 A Simple Goal Programming Model

This following implements a goal programming model solved using the simulated annealing (SANN) optimization algorithm in optim(). The model seeks the optimal harvest allocation among three forest management units by minimizing a weighted objective function that balances deviations from target timber production (100,000 m³), revenue ($5,000,000), and habitat quality (300 quality habitat units). Relative priorities are assigned through weights, with revenue receiving the highest relative priority, followed by timber production and habitat conservation. Harvest, revenue, and habitat are calculated from unit-specific coefficients, and each goal deviation is normalized by its target value (abs(harvest - target.harvest)/target.harvest) before being combined into a single weighted objective function. To ensure feasible solutions, large penalty terms are added whenever constraints are violated, including a maximum total harvested area of 200 ha, a minimum habitat requirement of 250 units, and a maximum harvest of 80 ha in Unit 3. The optimization begins from an initial allocation of 50 ha in each unit and uses simulated annealing with an initial temperature of 50 over 50,000 iterations to explore alternative harvest allocations and identify a near-optimal solution. After convergence, the script calculates the resulting harvest, revenue, habitat value, and normalized goal deviations, verifies that all constraints have been satisfied, and reports the optimal harvest allocation and associated performance measures.


set.seed(55)

############################################################
# MANAGEMENT GOALS
############################################################

# Target timber production (m3)
target.harvest <- 100000

# Target revenue ($)
target.revenue <- 5000000

# Target habitat score
target.habitat <- 300


############################################################
# GOAL PRIORITIES (WEIGHTS)
#
# Larger values indicate greater importance.
#
# Harvest > Revenue > Habitat
############################################################

w1 <- 3     # Harvest priority
w2 <- 5     # Revenue priority
w3 <- 1     # Habitat priority


############################################################
# FOREST MANAGEMENT UNIT COEFFICIENTS
#
#                 Unit1 Unit2 Unit3
# Timber (m3/ha)     400    800     600
# Revenue ($/ha)    20000 40000 30000
# Habitat Quality     1     2     4
############################################################

timber.coef  <- c(400, 800, 600)
revenue.coef <- c(20000, 40000, 30000)
habitat.coef <- c(1, 2, 4)

4.2.1 The Objective Function

This function defines the objective function for a goal programming optimization model that evaluates a proposed harvest allocation vector (x). It first calculates three performance measures—total timber harvest, revenue, and habitat—by multiplying the decision variables by their respective coefficient vectors and summing the results. It then computes the normalized deviation of each measure from its target value, expressing how far the solution is from meeting each objective relative to the target. These deviations are combined into a single weighted objective value (z) using the user-defined weights (w1, w2, and w3), allowing the relative importance of each goal to be specified. The function also enforces operational constraints by applying large penalty values whenever the total harvested area exceeds 200 ha, habitat falls below 250 units, or harvest in Unit 3 exceeds 80 ha. The penalties are added to the objective value, making infeasible solutions much less desirable. Finally, the function returns the total objective value, where lower values represent solutions that better satisfy the management goals while remaining within the operational constraints.

4.2.2 Code Notes:

4.2.2.1 Normalized Deviations from the Target

Normalized deviations are used in goal programming to make different objectives—such as timber volume, revenue, and habitat—comparable on the same scale, since these variables are measured in different units and magnitudes. Without normalization, a large-scale variable (e.g., revenue in tens of thousands of dollars) would dominate the objective function, even if it is not the highest priority. By dividing each deviation by its target value, each goal is expressed as a relative (percentage-like) shortfall or surplus, allowing the model to fairly balance trade-offs across competing objectives based on their assigned weights rather than their raw numerical size.

4.2.2.2 Operational Constraints

This section of the function implements penalty functions to discourage solutions that violate the optimization constraints. The variable penalty is initialized to zero, meaning there is no penalty if all constraints are satisfied. Each if statement checks whether a constraint has been violated. If the total harvest area (sum(x)) exceeds 200 ha, a penalty equal to 1000 × (sum(x) - 200) is added, so the penalty increases in proportion to the number of hectares over the limit. Similarly, if the habitat value is less than the required minimum of 250 units, a penalty of 1000 × (250 - habitat) is added, with larger shortfalls receiving larger penalties. Finally, if the harvest allocated to Unit 3 (x[3]) exceeds 80 ha, another penalty of 1000 × (x[3] - 80) is applied. Because these penalties are added to the objective function, any solution that violates one or more constraints receives a much larger objective value, making it far less likely to be selected by the optimization algorithm. The multiplier of 1000 is chosen to make constraint violations much more costly than small improvements in the objective, effectively encouraging the algorithm to prioritize feasible solutions.




goal_function <- function(x) {
  
  ##########################################################
  # PERFORMANCE METRICS
  ##########################################################
  
  harvest <- sum(timber.coef * x)
  
  revenue <- sum(revenue.coef * x)
  
  habitat <- sum(habitat.coef * x)
  
  ##########################################################
  # NORMALIZED DEVIATIONS FROM TARGETS
  ##########################################################
  
  d.harvest <- abs(harvest - target.harvest) /
    target.harvest
  
  d.revenue <- abs(revenue - target.revenue) /
    target.revenue
  
  d.habitat <- abs(habitat - target.habitat) /
    target.habitat
  
  ##########################################################
  # GOAL PROGRAMMING OBJECTIVE Function
  ##########################################################
  
  z <- (w1 * d.harvest) +
    (w2 * d.revenue) +
    (w3 * d.habitat)
  
  ##########################################################
  #
  # Constraint 1:
  # Total harvest area <= 200 ha : upper, one-sided goal
  #
  # Constraint 2:
  # Habitat >= 250 units: lower one - sided goal
  #
  # Constraint 3:
  # Unit 3 harvest <= 80 ha: upper one-sided goal
  ##########################################################
  
  penalty <- 0
  
  # Constraint 1
  if (sum(x) > 200) {
    penalty <- penalty +
      1000 * (sum(x) - 200)
  }
  
  # Constraint 2
  if (habitat < 250) {
    penalty <- penalty +
      1000 * (250 - habitat)
  }
  
  # Constraint 3
  if (x[3] > 80) {
    penalty <- penalty +
      1000 * (x[3] - 80)
  }
  
  ##########################################################
  # OBJECTIVE + PENALTIES
  ##########################################################
  
  z <- z + penalty
  
  return(z)
}

4.3 The SANN Optimization

This code uses simulated annealing to search for a harvest allocation that minimizes the goal_function while balancing multiple management objectives and discouraging constraint violations. The optim() function is called with the "SANN" method, starting from an initial solution of 50 ha in each management unit (par = c(50, 50, 50)) and running for up to 50,000 iterations with an initial temperature of 50.

The temperature (temp) parameter in simulated annealing controls how willing the algorithm is to accept worse solutions during the search. At the beginning of the optimization, a higher temperature makes the algorithm more likely to accept moves that increase the objective function (i.e., produce temporarily worse solutions). This helps the search escape local optima and explore a wider range of possible solutions. As the optimization progresses, the temperature is gradually reduced according to the simulated annealing schedule (internal to the optim function), making the algorithm increasingly selective and more likely to accept only improvements or very small deteriorations. In this code, temp = 50 specifies the initial temperature, providing a balance between exploration and exploitation during the early stages of the search. If the initial temperature is set too low, the algorithm may become trapped in a local optimum, whereas a very high temperature can result in excessive random exploration and slower convergence to a good solution.

Its important to note that the algorithm will always run until it reaches maxit iterations (unless an error or user interruption occurs). It does not have a convergence criterion (A convergence criterion is a predefined rule that determines when an optimization algorithm should stop because further iterations are no longer producing meaningful improvements in the objective function or solution. ) or early stopping rule based on reaching a minimum objective value. The trace perimeter is set to true to display the objective value and iteration interval.

4.3.1 Code Notes:

4.3.1.1 EXTRACT BEST SOLUTION FOUND

After the optimization is complete, the best solution found is extracted from result$par and used to calculate the final performance measures for timber harvest, revenue, and habitat. The code then computes the normalized deviations of these outcomes from their target values, allowing the user to assess how closely each objective was achieved.

4.3.1.2 CHECK CONSTRAINTS

Finally, it evaluates whether the optimized solution satisfies the three operational constraints by checking that the total harvested area does not exceed 200 ha, habitat remains at or above 250 units, and harvest in Unit 3 does not exceed 80 ha. These calculations provide a summary of both the quality and feasibility of the optimized solution.



result <- optim( #base R
  par = c(50, 50, 50),   # Initial solution of 50 ha for each unit
  fn = goal_function, #load the objective function
  method = "SANN", #use the simulated annealing method
  control = list(
    maxit = 50000, #interations
    temp = 50, #set initail tempture to 50, decrementing cooling schedule internal to the function
    trace = TRUE # report objective value
  )
)

4.4 The Results

The goal programming model identified an optimal harvest allocation of 38.61 ha in Unit 1, 77.36 ha in Unit 2, and 37.78ha in Unit 3, producing approximately 99997.82 m³ of timber, $4,999,890.88 in revenue, and 344.43 habitat quality units. All management constraints were satisfied, including the total harvested area remaining below the 200 ha limit, habitat exceeding the minimum target of 250 units, and harvesting in Unit 3 remaining below the 80 ha maximum. The deviations from the harvest, revenue, and habitat goals were all extremely small, indicating that the solution closely matched the desired management targets while balancing competing objectives. The weighted goal score of 0.148288, which is approaching zero, demonstrates that the optimization achieved an good compromise among timber production, economic return, and habitat conservation, with only negligible departures from the specified goals and no constraint violations. While Harvest and Revenue met thier targets (0 deviation) the proportional error relative to the target for Habitat Quality Units is 0.1481 reflective of their relative weights of importance.
